1) Scope & Goals

Enable sponsors and CROs to design, validate, and deploy eCRF forms for clinical trials.

Allow UAT (User Acceptance Testing) before promoting forms into production.

Support site-level data entry by CRC/PI with strong validation and discrepancy tracking.

Provide query management workflows for auto-generated and manual queries.

Ensure 21 CFR Part 11 compliance with audit trails, e-signatures, and role-based access.

Enable phase-based control: Start-up → Conduct → Close-out.

Deliver final locked, auditable datasets for statistical analysis.

2) Primary Roles

Study Designer (Sponsor/CRO): Creates forms, edit checks, and manages promotion.

CRC (Clinical Research Coordinator): Performs data entry at hospital/site.

PI (Principal Investigator): Performs/approves data entry; responsible for responses.

Data Manager (DM): Reviews data, raises queries, ensures data quality; read-only for entry.

CRA/Monitor: Reviews source data, raises manual queries.

Study Admin: Manages system, environments, roles, permissions.

Read-only Stakeholder: Auditor, regulator, sponsor for compliance checks.

3) Study Phases & Access Permissions
3.1 Start-up Phase

CRC/PI: Enter subject/visit data and respond to queries.

DM: Can view data but cannot edit.

Study Admin: Configure study, roles, and forms.

3.2 Conduct Phase

CRC/PI: Continue entering subject/visit data.

DM: Cannot edit data, but can raise queries and monitor data quality.

CRA/Monitor: Raise manual queries.

Study Admin: Maintain permissions and access.

Compliance Rule: Only CRC/PI may edit data fields (per 21 CFR).

3.3 Close-out Phase

CRC/PI: Finalize and lock forms.

DM: Review and confirm query closure, extract datasets.

Study Admin: Lock database and revoke user access.

System Output: Export final datasets (CDISC formats + error log + audit trail).

5) Functional Requirements
5.1 Form Builder (eCRF Designer)

Create study, site, subject visit schedules (e.g., Screening, Baseline, Follow-up).

Add field types: text, integer, decimal, date, dropdown, radio, checkbox, multi-select, file upload, calculated fields.

Configure attributes: datatype, length, required flag, default value, permissible range, regex validation, units.

Define cross-field logic: conditional required fields, derived/calculated fields.

Rules/Edit checks:

Datatype mismatch (numeric field cannot contain alphabets).

Range checks (temperature 34–43 °C).

Cross-field validation (systolic > diastolic).

Cross-visit validation (weight change > 30% from baseline).

Versioning: Draft → UAT → Production.

Preview and test with sample data.

5.2 UAT Workspace

Test environment for validating forms and edit checks.

Roles: CRC/PI simulate entry; DM monitors and verifies rules.

UAT Defect Workflow: Open → In Progress → Fixed → Re-tested → Closed.

Capture traceability: requirement → rule → test case → defect → fix evidence.

Export UAT evidence (PDF/ZIP) for audit.

5.3 Release & Promotion Management

Controlled release from UAT → Production.

Requires e-signature and time-stamped promotion logs.

Change impact report: schema diff, rules diff.

Hotfix workflow for urgent changes.

Immutable version history.

5.4 Site Data Entry

Subject lifecycle: Screened → Enrolled → Completed/Withdrawn.

Visit schedule management (planned, missed, unscheduled visits).

CRC/PI enter data in production forms with:

Real-time validation (client-side + server-side).

Error popup if data type mismatch occurs (e.g., entering “ABC” in temperature field).

Autosave, draft, and final submission modes.

Save states: In Progress → Submitted → Verified → Locked.

Attach source documentation (optional).

5.5 Validation & Edit Check Engine

Supports synchronous validation (inline checks) and asynchronous validation (batch checks across visits).

Severity levels:

Hard Error (must fix).

Warning (allow save but flagged).

Note (informational).

Multi-language error messages.

Supports unit normalization (e.g., °C vs °F).

5.6 Discrepancy & Query Management

Auto-discrepancies: Triggered by failed edit checks.

Manual queries: Raised by DM, CRA, or CRC/PI.

Each discrepancy record contains:

Study, Site, Subject, Visit, Form, Field, Value entered, Expected Value, Severity, Message, Owner, Status.

Workflow:

Open → Assigned → Site Answered → DM Reviewed → Resolved → Closed.

Accepted as Valid: Discrepancy marked clinically valid, stored in Error Log with justification.

SLA reminders for open discrepancies.

Export discrepancy tables (CSV/XLSX/PDF).

5.7 Audit Trail & Compliance

Track all insert/update/delete actions with: user, role, timestamp, IP, old/new values.

Mandatory reason for change on edits.

E-signatures for:

Form release promotion.

Data lock.

Final data export.

Audit logs must be immutable and exportable for regulators.

5.8 Dashboards & Reports

Study Metrics: enrollment progress, visit completion rates.

Data Quality: discrepancy counts, SLA breaches, open vs closed queries.

Validation Failures: top failing rules by frequency.

Query Aging: open queries grouped by age buckets.

Export reports in CSV, XLSX, PDF formats.

5.9 Security & Administration

RBAC (Role-based Access Control) for Start-up, Conduct, Close-out phases.

SSO (SAML/OIDC), MFA.

Environment segregation (DEV/UAT/PROD).

Data encryption: AES-256 at rest, TLS 1.2+ in transit.

Compliance: GCP, 21 CFR Part 11, GDPR.

5.10 Close-out & Data Export

Database lock: no edits allowed after locking.

CRC/PI access revoked post lock.

DM/Study Admin export datasets in:

CDISC SDTM/ADaM for regulatory submission.

Raw CSV/XML with error logs.

Full audit trail export.

Immutable datasets delivered to statistical analysis team.
